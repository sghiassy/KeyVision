# KeyVision iOS App

A native iPhone app that recognizes keys in real time — hold a key in front of the camera and see it labeled instantly, with no network round-trip in the hot path.

---

## How it works

Recognition happens entirely on-device using a CoreML-converted DINOv2 model. The live camera feed is processed at ~2 fps; each frame is segmented, embedded, and compared against locally-stored embeddings using cosine similarity. When a match is found, a label chip appears over the camera view.

Keys enrolled on the device are synced bidirectionally with the backend so the web admin at `/admin` remains a valid management interface.

```
Camera frame
  └─ Segmenter      Vision VNGenerateForegroundInstanceMaskRequest → 224×224 crop
  └─ Embedder       CoreML DINOv2 ViT-B/14 → 768-d float32 L2-normalized vector
  └─ RecognitionEngine  vDSP dot-product × stored embeddings → ranked MatchResult[]
  └─ CameraController   publishes top result → RecognitionOverlay renders label chip
```

## Architecture

### Project structure

```
ios/
├── project.yml                     # xcodegen spec — source of truth for the Xcode project
├── KeyVision.xcodeproj             # generated by xcodegen; do not hand-edit
└── KeyVision/
    ├── KeyVisionApp.swift          # @main entry point; triggers sync on launch
    ├── CoreML/
    │   └── DINOv2.mlpackage        # ~165 MB FLOAT16 CoreML model (generated once, tracked via Git LFS)
    ├── Models/
    │   ├── KeyModel.swift          # Key struct (Identifiable, Codable)
    │   └── EmbeddingEntry.swift    # EmbeddingEntry: image_id, key_id, label, [Float]
    ├── Services/
    │   ├── LocalStore.swift        # SQLite (raw sqlite3) — keys + embeddings, thread-safe
    │   ├── APIService.swift        # URLSession wrappers for all backend endpoints
    │   ├── SyncService.swift       # Bidirectional sync: GET /sync pull + POST /embedded push
    │   ├── Segmenter.swift         # Vision iOS 17+ foreground segmentation → 224×224 CGImage
    │   ├── Embedder.swift          # CoreML DINOv2 wrapper → [Float] 768-d
    │   └── RecognitionEngine.swift # Segment → embed → match; pure, no UI dependencies
    ├── Camera/
    │   ├── CameraController.swift  # AVCaptureSession, every-15th-frame recognition, @Published result
    │   └── CameraPreviewView.swift # UIViewRepresentable wrapping AVCaptureVideoPreviewLayer
    └── Views/
        ├── RootView.swift                      # TabView: Camera | Keys | Settings
        ├── Camera/
        │   ├── CameraTabView.swift             # Full-screen preview + overlay
        │   └── RecognitionOverlay.swift        # Label chip (green=high, yellow=maybe)
        ├── Keys/
        │   ├── KeysListView.swift              # All enrolled keys with sync badges
        │   └── KeyDetailView.swift             # Thumbnails + delete
        ├── Enrollment/
        │   ├── EnrollmentFlowView.swift        # 3-step guided capture → segment → embed → sync
        │   └── CaptureStepView.swift           # Single capture step with preview/confirm/retake
        └── Settings/
            └── SettingsView.swift              # Server URL, Sync Now, last-sync timestamp
```

### Key design decisions

| Decision | Rationale |
|---|---|
| On-device segmentation (Vision) instead of server-side OpenCV | iOS 17 `VNGenerateForegroundInstanceMaskRequest` is ML-based and significantly more robust than Canny edges on varied backgrounds |
| CoreML FLOAT16 | Halves model size (~330 MB → ~165 MB) with negligible accuracy loss at inference |
| Device is embedding authority | iOS computes embeddings via CoreML and pushes them to the server. The server stores them without re-running DINOv2, ensuring both sides share the same embedding space |
| Raw SQLite3 (no ORM) | Avoids adding a dependency (e.g. GRDB) for a simple two-table schema |
| vDSP dot product | SIMD-accelerated; ~microseconds for 768-d vectors at 2 fps |
| Every-15th-frame processing | ~2 recognition passes/sec at 30 fps — fast enough for real-time feel while keeping CPU/battery reasonable |

### Confidence thresholds

These mirror the server's `matcher.py` constants:

| Level | Similarity | UI |
|---|---|---|
| `high` | ≥ 0.85 | Green chip + "View Details" button |
| `maybe` | 0.65 – 0.85 | Yellow chip, no button |
| `no_match` | < 0.65 | No overlay |

Chip auto-hides after 2 seconds with no new high-confidence result.

### Sync protocol

**Server → device (pull):** `GET /sync` returns all keys with base64-encoded float32[768] embeddings. The app diffs against `LocalStore`, adds new keys/images, and removes deleted keys.

**Device → server (push):** After enrollment, the app calls `POST /keys/{key_id}/images/embedded` with the 224×224 JPEG crop and the raw embedding blob (3072 bytes, float32 little-endian). The server stores both without re-running DINOv2.

## Dev setup

**Requirements:**
- macOS 14+
- Xcode 15+
- iOS 17+ device (recommended for camera; simulator works for UI development but has no camera)
- [xcodegen](https://github.com/yonaskolb/XcodeGen): `brew install xcodegen`
- Python 3.9 venv with repo dependencies (only needed for the one-time CoreML conversion)

### 1. Generate the Xcode project

The `.xcodeproj` is generated from `ios/project.yml` and is not committed. Run this once after cloning and any time you add or remove Swift files:

```bash
cd ios
xcodegen generate
```

### 2. Convert DINOv2 to CoreML (one-time)

This downloads the DINOv2 weights (~330 MB) and produces `ios/KeyVision/CoreML/DINOv2.mlpackage` (~165 MB). It runs a validation step that asserts CoreML and PyTorch produce embeddings within ε = 1e-4 on 5 random images.

```bash
# from root
source .venv/bin/activate
pip install coremltools  # if not already installed
python scripts/convert_dinov2_to_coreml.py
```

Expected output:
```
Loading facebook/dinov2-base from HuggingFace...
Baking position embeddings for 224×224 inputs...
  Interpolating position embeddings: 1369 patches → 256 patches (37×37 → 16×16)...
  Done. New position_embeddings shape: torch.Size([1, 257, 768])
Tracing model with torch.jit.trace...
Converting to CoreML FLOAT16...
Saved: ios/KeyVision/CoreML/DINOv2.mlpackage
Validating on 5 random images...
  Image 1: cos_sim = 0.999983  max |delta| = 0.001394
  ...
Min cosine similarity : 0.999971 (threshold: >= 0.999)
Max element-wise delta: 0.001480   (threshold: <  0.005)
Validation PASSED.
```

> **Why two thresholds?** FLOAT16 introduces per-element quantization noise of ~0.001–0.002, which is expected. The element-wise delta of ~0.001 sounds large but the two embeddings remain nearly parallel (cosine similarity > 0.9999), which is what actually matters for recognition. The primary validation gate is cosine similarity ≥ 0.999; the element-wise check (< 0.005) is a secondary sanity check.

> **Position embedding note:** `facebook/dinov2-base` was pretrained at 518×518. Its stored position embeddings cover 1369 patches (37×37), but we feed 224×224 images (256 patches, 16×16). The script pre-computes the bicubic interpolation once in Python and bakes the result into the model as a constant before tracing — eliminating the `upsample_bicubic2d` op that coremltools does not support.

The `.mlpackage` is excluded from git via `.gitignore`. Track it with Git LFS if you want to commit it:

```bash
git lfs track "ios/KeyVision/CoreML/DINOv2.mlpackage"
git add .gitattributes
```

### 3. Set your Team ID

Open `ios/project.yml` and set `DEVELOPMENT_TEAM` to your Apple Developer Team ID, then re-run `xcodegen generate`. You can find your Team ID in Xcode → Settings → Accounts, or at [developer.apple.com](https://developer.apple.com).

```yaml
settings:
  base:
    DEVELOPMENT_TEAM: "XXXXXXXXXX"   # ← your 10-character Team ID
```

### 4. Run on device

1. Connect your iPhone (iOS 17+)
2. Open `ios/KeyVision.xcodeproj` in Xcode
3. Select your device as the build target
4. Build and run (`⌘R`)
5. In the **Settings** tab, set the server URL to your Mac's local IP:
   ```
   http://192.168.x.x:8000
   ```
   (Find it with `ifconfig | grep "inet 192"`)

The app syncs keys from the server on launch. If no server is running, enrollment and recognition still work fully offline using whatever keys are already in `LocalStore`.

## Running the backend alongside the app

```bash
source .venv/bin/activate
uvicorn app.main:app --host 0.0.0.0 --reload
```

The `--host 0.0.0.0` flag makes the server reachable from your iPhone on the same Wi-Fi network.

Verify the new sync endpoint:

```bash
curl http://localhost:8000/sync   # returns [] if no keys enrolled yet
```

## Deploying to production (TestFlight / App Store)

> For Phase 0 (internal testing), direct install via Xcode or TestFlight is sufficient. The notes below apply when submitting to App Store Connect.

### App Store size

The bundled DINOv2.mlpackage is ~165 MB. Apple's over-the-air install limit is 200 MB, so bundling is fine for now. If the compressed IPA exceeds the limit, consider:

- Switching to `facebook/dinov2-small` (~40 MB CoreML, 384-d embeddings — requires re-enrolling all keys and updating embedding dimension constants)
- Using [On-Demand Resources](https://developer.apple.com/library/archive/documentation/FileManagement/Conceptual/On_Demand_Resources_Guide/) to download the model after first launch

### Build for distribution

1. Set the scheme to **Release** in Xcode
2. Product → Archive
3. Upload to App Store Connect via Xcode Organizer or `xcrun altool`

### Production server URL

In the **Settings** tab, users enter the server URL. The production backend is at `https://keyvision.cloud`. There is no hardcoded default — users must set this once after install.

## Troubleshooting

**"No foreground subject detected" during enrollment**
The Vision segmenter needs a reasonably distinct subject. Use a plain light background and make sure the key fills at least 5% of the frame.

**Embeddings from device don't match server**
Run the conversion script's validation step (`python scripts/convert_dinov2_to_coreml.py`) — it will catch any preprocessing mismatch between CoreML and PyTorch.

**Xcode "No such module 'DINOv2'"**
The CoreML model must be present at `ios/KeyVision/CoreML/DINOv2.mlpackage` before building. Run the conversion script first, then regenerate the Xcode project with `xcodegen generate`.

**Recognition too slow / battery drain**
Reduce frame processing rate in `CameraController.swift` by increasing `frameInterval` (default: 15). Setting it to 30 halves the recognition rate to ~1 fps.
